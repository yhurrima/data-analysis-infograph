# 图表叙事家

## 1. 作品定位

**图表叙事家** 是一个面向分析报告、研究总结、业务复盘和传播场景的 Wan Skill。

它的核心不是固定生成某几种图表，而是：

- 先识别报告里有什么数据形态
- 再决定适合讲什么故事
- 最后按叙事逻辑组装信息图模块

相比“固定 7 模块”的方案，图表叙事家更适合处理真实世界里结构不统一的报告。  
如果一份报告没有“结构对比”类内容，就不强行塞入该模块；如果一份报告有明显的人群、流程或原因分析，则优先转成对应模块。

---

## 2. 核心思路

整体流程：

`用户输入报告 -> 提取数据点与文本信号 -> 识别数据形态 -> 生成候选模块池 -> 组装标准版/高密度版 -> 构建 Prompt -> Wan 生图`

关键变化有两点：

1. **先判断内容，再选模块**
   不再预设每份报告都必须有相同的模块组合。

2. **按输出密度组装**
   用户可在两种方案中选择：
   - **标准版**：5 个模块
   - **高密度版**：8 个模块

## 2.2 调用前确认

在正式生成前，这个 skill 应先完成用户确认，而不是默认直接生图。

### 必须确认的输入

1. **报告内容**
   - 报告原文
   - 或报告链接
   - 或结构化摘要

2. **输出密度**
   - `标准版`：5 个模块
   - `高密度版`：8 个模块

3. **目标风格**
   - `minimal`
   - `vibrant`
   - `dark`
   - `academic`

### 交互规则

- 如果用户没有提供报告内容，不能开始生成
- 如果用户没有指定密度，必须先确认是 `标准版` 还是 `高密度版`
- 如果用户没有指定风格，必须先确认四种风格中的哪一种
- 只有当报告内容、密度、风格这三项都明确后，才进入模块选择和 prompt 生成

### 推荐确认话术

```text
在正式生成前，请先确认三项内容：
1. 你的报告内容或报告链接是什么？
2. 你要标准版（5模块）还是高密度版（8模块）？
- 标准版：信息更精炼，重点更突出，适合汇报扫读、快速理解，文字更少、可读性更高。
- 高密度版：信息更丰富，覆盖内容更多，适合一图读懂、完整展示和传播，但画面会更满一些。

3. 你希望的风格是哪一种？

| 风格名称 | 风格代号 | 简介 | 适合场景 |
| --- | --- | --- | --- |
| 简约现代 | minimal | 白色背景，黑白为主，黄色点缀，整体干净克制、留白更多 | 清爽传播、简洁展示 |
| 活泼明亮 | vibrant | 颜色更鲜明，视觉更有冲击力，整体更有传播感 | 社媒传播、视觉吸引 |
| 深色科技 | dark | 深色背景，高对比度，偏科技感和未来感 | 趋势主题、科技内容 |
| 学术论文 | academic | 更正式、更规整，像研究报告或论文图表页 | 评审展示、专业汇报 |

你可以直接回复我两项选择，例如：
- 标准版 + 简约现代
- 高密度版 + 学术论文
```

### 默认生成配置

如果用户没有特别指定模型或分辨率，默认使用：

- 模型：`wan2.7-image-pro`
- 分辨率：`1774*2364`（3:4 竖版）

这个默认值与当前 skill 的标准输出保持一致，适合移动端阅读和信息图展示。

### 可直接跳过确认的情况

如果用户已经明确说出：

- 使用哪份报告
- 使用哪种密度
- 使用哪种风格

则不需要重复询问，直接进入后续流程。

## 3. 输出版本

### 标准版

- 模块数量：**5 个**
- 适用场景：汇报扫读、管理层快速浏览、信息密度适中
- 组成逻辑：`3 个必选模块 + 2 个动态模块`

### 高密度版

- 模块数量：**8 个**
- 适用场景：移动端长图、研究内容压缩、一图读懂
- 组成逻辑：`3 个必选模块 + 5 个动态模块`

---

## 4. 模块策略

### 4.1 必选模块

以下模块默认保留，保证任何报告都能形成完整叙事闭环：

1. **报告速览**
   标题、时间范围、研究对象、核心标签

2. **核心发现**
   用一句主结论加 1-2 条数据支撑，说明“这份报告最重要的发现是什么”

3. **一句话结论**
   用一句总结收尾，并附上数据来源或研究背景

### 4.2 动态候选模块池

其余模块按报告内容动态选择，初始候选池建议包括：

- **趋势分析**
  适合时间序列、同比、环比、增长、下降、波动

- **结构对比**
  适合占比、构成、分布、份额、类别拆分

- **排名洞察**
  适合 TOP 榜单、领先项、头部集中、城市/品牌/品类排序

- **关键指标卡片**
  适合 KPI 多、需要快速扫读时

- **用户画像**
  适合年龄、性别、城市、客群、消费层级等人群特征

- **流程拆解**
  适合转化路径、用户旅程、步骤流程、阶段分析、漏斗

- **原因归因**
  适合驱动因素、影响因素、增长原因、下滑原因

- **行动建议**
  适合报告中出现明确建议、策略方向、下一步动作

## 5. 数据形态识别规则

在生成模块前，先把报告拆成两类信号：

- **数值型信号**：数值、百分比、单位、同比环比、时间点
- **语义型信号**：排名、原因、建议、用户分层、流程描述

### 5.1 识别维度

| 数据形态 | 识别信号示例 | 对应优先模块 |
|---------|-------------|-------------|
| 时间趋势 | Q1/Q2、1月-12月、同比、环比、增长、下降 | 趋势分析 |
| 占比结构 | 占比、份额、构成、分布、XX% | 结构对比 |
| 排名排序 | TOP、排名、第1、第2、领先 | 排名洞察 |
| 核心 KPI | DAU、GMV、留存率、转化率、客单价 | 关键指标卡片 |
| 人群特征 | 用户、年龄、城市、性别、客群、偏好 | 用户画像 |
| 路径流程 | 流程、步骤、阶段、漏斗、旅程、转化路径 | 流程拆解 |
| 原因机制 | 原因、驱动、影响、带动、受限于 | 原因归因 |
| 策略结论 | 建议、策略、动作、优化、下一步 | 行动建议 |

### 5.2 判定原则

- 若报告同时存在多种信号，优先选择信息价值更高、且数据更完整的模块
- 若某模块只有标题性描述、缺少数据支撑，则降低优先级
- 若模块内容与核心结论弱相关，则不进入最终版面
- 若动态候选不足，则用“关键指标卡片”补足模块数量

## 6. 模块组装逻辑

### 6.1 排序思路

先为每个候选模块打分，再按分数排序：

- 数据完整度
- 与核心结论的相关度
- 可视化表达清晰度
- 是否能形成信息层次

示例优先级：

```text
score = 数据完整度 * 0.4
      + 结论相关度 * 0.3
      + 可视化适配度 * 0.2
      + 阅读增益 * 0.1
```

### 6.2 组装规则

**标准版 5 模块**

- 固定保留：报告速览、核心发现、一句话结论
- 从候选模块池中选择分数最高的 2 个

**高密度版 8 模块**

- 固定保留：报告速览、核心发现、一句话结论
- 从候选模块池中选择分数最高的 5 个

### 6.3 兜底规则

如果候选模块不够：

- 优先补 `关键指标卡片`
- 其次允许拆出一个更细化的模块，例如：
  - `趋势分析` 拆成 `总体趋势` + `关键拐点`
  - `用户画像` 拆成 `基础画像` + `行为偏好`

如果报告偏定性，没有足够图表数据：

- 允许转成“结论卡片 + 原因归因 + 行动建议”型版式
- 不强求必须出现饼图、柱状图、排名条

## 7. 中间结构化方案

建议在正式生图前，先输出一份中间结构化结果，便于人工复核：

```yaml
title: 2026年Q1用户增长分析
core_insight: 用户增长势头延续，复购率提升是增长质量改善的关键
edition: standard
required_modules:
  - 报告速览
  - 核心发现
  - 一句话结论
candidate_modules:
  - name: 趋势分析
    score: 0.92
    evidence:
      - Q1-Q4 留存率变化
      - 同比提升 6%
  - name: 用户画像
    score: 0.84
    evidence:
      - 一线城市用户占比 58%
      - 25-34 岁用户为主力
selected_modules:
  - 报告速览
  - 核心发现
  - 趋势分析
  - 用户画像
  - 一句话结论
data_source: 2026年Q1增长分析报告
style: vibrant
```

这个中间层的价值：

- 让模块选择过程可解释
- 便于核对模块是否合理
- 便于比赛展示“产品逻辑”而不只是生图结果

## 8. Prompt 构建原则

Prompt 不再写死“7 模块结构”，而是改成：

- 明确总模块数：5 或 8
- 明确本次实际入选的模块列表
- 每个模块只填对应数据，不强行补位

Prompt 中必须体现：

- 版本类型：标准版 / 高密度版
- 阅读顺序：从上到下、从主结论到细分证据
- 每个模块的标题、数据、洞察、推荐图形元素
- 文字长度限制，避免图中过度拥挤

### 8.1 Prompt 编写规范

写 prompt 时，不要只把报告原文切成若干段直接堆给模型。更稳的方式是：

1. 先输出一个中间结构化方案
2. 再把结构化方案翻译成图像模型可理解的视觉 prompt
3. 最后再补风格约束和文字安全约束

建议的写法顺序：

- `Main Title / Subtitle / Source / Period`
- `Edition`：standard 或 dense
- `Module List`：本次实际模块
- `Module Content`：每个模块单独写字段
- `Style Direction`
- `Graphic Requirements`
- `Text Safety`

### 8.2 模块字段化规则

为了减少中文信息图中“标题乱入”“正文被误识别为副标题”的问题，模块内容尽量字段化，而不是整句散写。

更稳的写法示例：

```text
Module 2 核心发现:
- 主数字: 768.0万对
- 变化标签: +12.4%
- 短结论: 2023年结婚登记止跌回升
- 补充说明: 结束连续多年下滑趋势
```

不够稳的写法示例：

```text
Module 2 核心发现:
- 2023年结婚登记 768.0万对，同比增长12.4%
```

后者容易让模型把“2023年结婚登记”误判成新的小标题。

### 8.3 文字安全规则

以下规则应优先加入 prompt：

- 每个模块只允许一个可见标题
- 不要在模块内部再创建二级标题
- 主数字、趋势标签、短结论分开写
- 句子尽量短，优先用短标签替代长说明
- 模块内避免超过 3-4 条长句

### 8.4 版本差异规则

**标准版**

- 优先保信息清晰度
- 适合短结论、核心趋势、核心对比
- 更适合避免中文文字糊掉

**高密度版**

- 优先保信息覆盖度
- 适合展示方法能力和多模块编排
- 更容易出现文字拥挤和标题误判，需要更强字段化写法

### 8.5 风格调用规则

风格 prompt 不要临场即兴发挥，优先使用风格指南里的固定锚点词和禁忌词。

- `minimal`：强调白底、黑白主色、黄色唯一强调、留白、自由流动布局
- `vibrant`：强调明亮但专业、紫红黄配色、强编辑感
- `dark`：强调深色背景、高对比霓虹强调、科技感
- `academic`：强调学术规范、精确对齐、正式展示、克制强调色
- 所有风格都不应包含平台 logo、平台 UI、角标或水印式品牌元素

更具体的写法参见 [chart-storyteller-style-guide.md](/Users/yhurri/Desktop/Codex_workfile/tasks/wanxiang-skill-competition/data-analysis-infograph-skill/chart-storyteller-style-guide.md)

## 9. 伪代码示意

```python
REQUIRED_MODULES = ["报告速览", "核心发现", "一句话结论"]

CANDIDATE_MODULES = [
    "趋势分析",
    "结构对比",
    "排名洞察",
    "关键指标卡片",
    "用户画像",
    "流程拆解",
    "原因归因",
    "行动建议",
]

def build_infographic(report_text: str, edition: str):
    signals = detect_data_shapes(report_text)
    candidates = score_candidate_modules(signals, report_text)

    if edition == "standard":
        dynamic_count = 2
    elif edition == "dense":
        dynamic_count = 5
    else:
        raise ValueError("unknown edition")

    selected_dynamic = select_top_modules(candidates, dynamic_count)
    selected_modules = REQUIRED_MODULES + selected_dynamic

    return {
        "edition": edition,
        "selected_modules": selected_modules,
        "prompt": render_prompt(selected_modules, report_text),
    }
```

## 11. 注意事项

1. 不要把“模块数量固定”误当成“信息图质量稳定”的来源，真正稳定的是模块选择逻辑。
2. 动态模块不是越多越好，标准版要优先保证阅读顺序和核心结论清晰。
3. 当报告缺乏结构型数据时，允许更多使用结论卡片和归因模块。
4. 当报告数据维度很多时，高密度版也要限制单模块字数，避免生成图文字糊成一团。
5. 输出前建议先给用户展示模块清单，确认“讲故事的方向”是否正确，再正式生图。
6. 如果风格效果不稳定，先检查风格锚点词是否写偏，再考虑换模型或改尺寸。
7. 如果出现“标题乱入”，优先检查模块字段写法，而不是立刻归因到模型随机性。
8. 如果出现平台 logo、平台角标或 app UI 元素，优先检查 prompt 中是否误写了平台名称，并补充显式的负向约束。
